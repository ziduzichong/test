# 架构审查报告

## 总体评价

整体架构设计清晰务实，选择 Node.js + SQLite + 原生前端的技术栈与"小团队零运维"的需求高度匹配。模块划分合理，接口遵循 RESTful 规范，数据流单向无循环依赖。主要风险集中在安全防护（XSS、CSRF、文件访问控制）和可维护性（单文件膨胀）方面，均为可修复的一般性问题。

## 各维度评分

| 维度 | 评分(1-5) | 评分理由 |
|------|----------|---------|
| 模块划分 | 4 | 6大模块职责单一，边界清晰。缺少统一错误处理模块和日志模块。 |
| 接口设计 | 4 | RESTful 语义清晰，但缺少分页参数和统一错误响应格式，bulk 操作下 GET 无分页可能性能瓶颈。 |
| 数据流 | 5 | 公开只读→认证读写，单向流动，无循环依赖。数据转换节点标注明确。 |
| 可扩展性 | 4 | 模块化 API 便于横向扩展新资源类型（如活动报名、作品投票）。但 server.js 单文件限制了纵向拆分。 |
| 可测试性 | 4 | API 接口独立可测，但缺少自动化测试框架规划，前端无测试策略。 |
| 可移植性 | 5 | Node.js + SQLite 跨平台，纯原生前端零框架依赖，部署仅需 node 和 npm。 |
| 性能 | 4 | better-sqlite3 同步驱动适合 <50 并发，但文件流输出未使用 stream.pipe 可能导致大文件内存溢出。 |
| 安全性 | 3 | 密码哈希和 httpOnly cookie 到位。但富文本 sanitize 方案未指定、public/uploads 直接暴露有绕过鉴权风险、缺少 CSRF 防御和速率限制。 |

## 发现的问题

| 编号 | 问题描述 | 严重程度 | 影响范围 |
|------|---------|---------|---------|
| P1 | 富文本 XSS 防护不明确——设计文档提到"做 sanitize"但未指定具体方案（DOMPurify/Sanitizer API？服务端过滤？），实现时可能遗漏 | 严重 | 所有公告渲染页面 |
| P2 | public/uploads/ 目录如果被 express.static 挂载，攻击者可绕过 API 鉴权直接下载文件 | 一般 | 文件管理模块 |
| P3 | 缺少 CSRF 保护——所有 POST/PUT/DELETE 仅依赖 session cookie，可能被跨站请求伪造攻击 | 一般 | 全部写操作 |
| P4 | 缺少登录速率限制——攻击者可通过暴力破解猜解密码 | 一般 | 认证模块 |
| P5 | 缺少统一错误响应格式——前端需针对不同错误格式做各种判断，维护成本高 | 一般 | 全部 API |
| P6 | 文件上传缺少类型校验——可上传 .exe/.sh/.bat 等可执行文件，存在服务器风险 | 一般 | 文件管理模块 |
| P7 | server.js 单文件设计——随着功能增加（预计 400+ 行），可维护性显著下降 | 建议 | 后端整体 |
| P8 | 缺少分页支持——公告>100条时一次性加载全部影响首屏响应时间 | 建议 | 公告/成员/文件列表 |

## 改进建议

1. **P1 严重**：前端集成 DOMPurify（CDN，约 15KB），渲染公告内容前对 HTML 做清理，去除 `<script>`、`onerror` 等危险标签/属性。服务端不做过滤（信任已登录开发者），仅前端防护。

2. **P2 一般**：文件存储移到 `data/uploads/`（非 public 目录），下载通过 `/api/files/download/:id` 代理流输出。移除 express.static 对 uploads 的挂载。

3. **P3 一般**：由于是内部小团队使用，CSRF 风险较低。短期可暂不处理（记录为已知限制），长期可加 csurf 中间件。

4. **P4 一般**：添加 express-rate-limit 中间件，登录接口限制 10次/分钟/IP。

5. **P5 一般**：统一响应格式 `{ ok: boolean, data?: any, error?: string }`，所有 API 按此范式返回。

6. **P6 一般**：Multer fileFilter 增加白名单校验，仅允许常见文档/代码/压缩包类型。

7. **P7 建议**：当 server.js 超过 300 行时，按模块拆分为 routes/auth.js、routes/announcements.js 等独立文件。

8. **P8 建议**：GET 列表接口增加 `?page=1&limit=20` 参数，默认 limit=50。

## 替代技术路线

| 路线 | 适用场景 | 优势 | 劣势 | 迁移成本 |
|------|---------|------|------|---------|
| SQLite → PostgreSQL | 并发 >100 或需要全文搜索公告时 | 更好的并发和搜索能力 | 需要额外部署和维护 PG 服务 | 中（改驱动 + SQL minor 差异） |
| 原生前端 → Vue 3 | 交互复杂度持续增长（超过 5 个管理页面）时 | 组件化提高复用、状态管理清晰 | 需要构建工具链，失去"零依赖"优势 | 高（重写全部前端） |

当前阶段不推荐切换，以上作为未来演进参考。

## 结论

架构设计整体符合小团队使用场景，无致命缺陷。P1（XSS防护）和P2（文件访问控制）需要在编码阶段优先修复，其余问题可在代码审查阶段逐一落实。建议通过后进入编码阶段。
